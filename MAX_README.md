# MAX Messenger API + maxapi (Python): подводные камни

Сводка для разработки ботов на **MAX** с библиотекой **[maxapi](https://pypi.org/project/maxapi/)** (сообщество, не официальный SDK). Официальная HTTP-документация: [dev.max.ru/docs-api](https://dev.max.ru/docs-api).

---

## 1. `MessageCallback.answer()` затирает свежий `message.edit()` (клавиатура «откатывается»)

**Симптом:** после нажатия inline-кнопки вы делаете `await event.message.edit(..., attachments=[новая_клавиатура])`, текст верный, но через доли секунды снова показывается **старая** клавиатура (или «мигание»).

**Причина (реализация maxapi):** если у callback есть исходное сообщение с `body`, метод `MessageCallback.answer()` **всегда** кладёт в ответ на `POST /answers` объект `message` с **`attachments` из исходного тела** (снимок на момент получения апдейта), даже если вы передали только `notification` и не меняли текст:

- Берётся `original_body.attachments` и подставляется в `MessageForCallback` (строки ~132–134 в `maxapi/types/updates/message_callback.py`).
- Запрос уходит **после** вашего `edit`, и клиент снова рисует старую inline-клавиатуру.

**Правильное подтверждение callback без перезаписи сообщения:** `message=None`, чтобы не слать старые вложения. При этом **пустой JSON** для `POST /answers` API не принимает — нужно хотя бы **`notification`** (иначе `400 proto.payload`: «`message` or `notification` required»). Используйте непустую строку (например один пробел `" "` — некоторые билды API отвергают zero-width символы), либо текст тоста.

```python
await bot.send_callback(
    callback_id=event.callback.callback_id,
    message=None,
    notification=" ",
)
```

В проекте: `app.callback_ack.send_callback_ack`.

Не вызывать «пустой» `event.answer()` после `edit`, если не хотите отката вложений.

**Когда `event.answer()` безопаснее:** если `message` или `body` в апдейте отсутствуют — уходит только `notification` / пустое тело без поля `message` (см. ветку `original_body is None` в том же файле).

---

## 2. `message.edit(attachments=None)` **не** очищает вложения

**Симптом:** ожидали убрать клавиатуру, передав `attachments=None`, а она осталась.

**Причина (maxapi):** в `Message.edit`, если передан `attachments is None`, а у сообщения уже есть `body.attachments`, библиотека **подставляет старые** вложения в запрос редактирования (`message.py`, условие ~502–507).

**Что делать:** явно передавать **пустой список**: `attachments=[]`, чтобы в `EditMessage` ушло `json["attachments"] = []` (пустой массив в PUT).

---

## 3. Редактирование сообщения через HTTP: пустые вложения

В `EditMessage.fetch()` тело запроса по умолчанию содержит `"attachments": []`; новые вложения добавляются только если список непустой. Пустой список соответствует «снять» inline-клавиатуру на стороне API (если платформа это поддерживает для вашего типа вложения).

---

## 4. Ответ на callback только с `notification`

Даже при `event.answer(notification="Текст")`, если `original_body` **есть**, maxapi всё равно отправляет `message` со **старыми** `attachments` (п. 1). Для сценария «показать тост и **не** трогать сообщение» безопаснее тот же `send_callback(..., message=None, notification="...")`.

---

## 5. Порядок роутеров в `Dispatcher`

Если есть «ловушка» вроде `@router.message_created()` без фильтра (все текстовые сообщения) и отдельный роутер с командами/FSM, **более специфичные** обработчики (админка, состояния) должны обрабатываться **раньше**. Иначе широкий хендлер перехватит апдейт.

Пример: `dp.include_routers(admin.router, user.router)` — сначала admin, потом user.

---

## 6. Проверка подписки на канал / чат

- Используются методы вроде `get_chat_member(chat_id, user_id)` и при необходимости обход `get_chat_members` постранично: иногда API ведёт себя нестабильно (пустой ответ там, где ожидали участника) — в проде имеет смысл fallback по списку участников (см. логику в проекте `user_is_channel_member`).
- Чтобы проверка имела смысл, **бот должен иметь права** в соответствующих чатах/каналах (как в требованиях MAX к боту в канале).

---

## 7. Идентификатор чата `chat_id`

У каналов/чатов в MAX `chat_id` часто **отрицательный**. Если сохранили id **без минуса**, `get_chat_by_id` может вернуть 404 — имеет смысл пробовать альтернативу с противоположным знаком (в проекте: `resolve_max_chat_id`).

---

## 8. Callback и отсутствие `message.body`

В callback `event.message` может быть, но `body` пустой — любые операции с `edit` недоступны. Имеет смысл проверять и отвечать через `send_callback` с `notification`, без `edit`.

---

## 9. `get_chat_by_link` (добавление канала по URL)

В maxapi: `GET /chats/{link}` — в путь подставляется последний фрагмент, извлечённый из строки регулярным выражением вида `@?[a-zA-Z]+[a-zA-Z0-9-_]*` (см. `GetChatByLink`). Если в ссылке нет подходящего фрагмента, будет `ValueError`. Бот должен **видеть** этот публичный чат через API, иначе — ошибка MAX.

---

## 10. Документация vs библиотека

- Контракт HTTP описан на [dev.max.ru/docs-api](https://dev.max.ru/docs-api); поведение **обёртки maxapi** (когда она подмешивает старые `attachments`, как трактуется `attachments=None`) нужно сверять с **исходниками** установленной версии пакета — это не всегда очевидно из docstring.

---

## Краткий чеклист после нажатия inline-кнопки

1. Обновили сообщение через `message.edit` — нужные `attachments` или `[]`.
2. Подтвердили callback через `send_callback_ack` / `message=None` + непустой `notification` (например один пробел).
3. Не вызывать `event.answer()` без осознанной необходимости, если только что меняли вложения через `edit`.
